# Enterprise RAG — 持续改进目标

> 本文档定义产品演进的分阶段目标与验收标准，作为长期迭代的方向盘。  
> **原则**：先内部用透、反馈闭环做扎实；架构上预留多租户与内外端分离，避免外网上线时推倒重来。  
> **执行方式**：按阶段顺序推进；每一具体改动开工前，必须过一遍文末「改动检查清单」。

---

## 一、总体愿景

| 阶段 | 定位 | 核心目标 |
|------|------|----------|
| **阶段 0** | 当前基线 | 单向 RAG 链路可用；trace / feedback 有钩子但未闭环 |
| **阶段 1** | 公司内部完美使用 | 反馈驱动持续改进；LLM 辅助研判；Admin 为运营中枢 |
| **阶段 2** | 外网 SaaS | 多租户隔离；平台端 / 客户管理端 / 客户使用端分级；合规与配额 |

```text
Collect → Triage(LLM) → Review Queue → Actuator → Evaluate →（回到 Collect）
```

**LLM 在反馈层的角色**：只做分类、建议、报告；**不直接无审批改生产**。执行走 Actuator 白名单 + 人工或策略审批。

## 二、非功能约束（每项改动必查）

每次动手前，在 PR / 任务描述中显式回答下列问题。任一项为「高风险且未缓解」则不应合并。

### 2.1 响应速率（Latency）

| 检查项 | 要求 |
|--------|------|
| 热路径 | `/chat`、`/chat/stream`、`/retrieve` **不得**同步增加额外 LLM 调用（反馈、评测、落库走异步） |
| 写入 | Feedback / trace 持久化：单次额外耗时目标 **< 10ms**（SQLite 单条或批量队列） |
| 检索 | 索引结构变更须跑 `scripts/bench_index_perf.py` 或等价基准，向量/BM25 P95 不明显退化 |
| 流式 | SSE 首 token 时间（TTFB）不因新中间件增加 **> 50ms** |
| 缓存 | 配置版本变更须 **失效检索缓存**（已有 `invalidate_search_cache` 模式） |

### 2.2 内存（Memory）

| 检查项 | 要求 |
|--------|------|
| 进程内 | 禁止在请求路径加载全量 feedback/trace 文件；列表查询必须分页 |
| 队列 | 后台 Triage 单次批处理有上限（建议 ≤ 50 条），避免 OOM |
| 模型 | 不为主路径新增常驻大模型；Triage 复用已有 `resolve_llm_runtime` |
| SQLite | 单库体积监控；历史 trace 归档策略（按月分表或 JSONL 冷存） |

### 2.3 稳定性（Stability）

| 检查项 | 要求 |
|--------|------|
| 失败隔离 | Feedback Loop 任何步骤失败 **不得** 导致 `/chat` 失败（try/except + 降级） |
| 幂等 | Actuator 动作可重试、可去重（如 `action_id` / 状态机） |
| 回滚 | 配置类变更必须版本化，支持一键回滚到上一 revision |
| 多 worker | 新状态存储须考虑 uvicorn `--workers N`（SQLite WAL 或迁 PostgreSQL 计划） |
| 依赖 | 新外部依赖须可选、可降级（与 RAGAS fallback 模式一致） |

### 2.4 可扩展性（Extensibility）

| 检查项 | 要求 |
|--------|------|
| 接口 | 新 API 走 `/api/v1/` 前缀规划；Schema 用 Pydantic，前后端 types 同步 |
| 插件 | Triage / Actuator 用注册表模式，新增动作不改核心循环 |
| 租户 | 新表/新字段预留 `tenant_id`，默认 `internal`，单租户行为不变 |
| 存储 | 抽象 `FeedbackStore` / `TraceStore`，便于阶段 2 换 PostgreSQL |

### 2.5 标准与规范（Standards）

| 检查项 | 要求 |
|--------|------|
| 测试 | TDD：核心路径 + 边界；改动须有 pytest / 前端单测 |
| 安全 | 管理类 API 鉴权；敏感字段不落日志；`deploy_security.md` 清单不倒退 |
| 可观测 | 新模块打结构化日志（含 `trace_id`、`tenant_id`）；关键计数可暴露 `/health` 或 debug |
| 文档 | 完成功能后更新 `README.md` 已实现功能表；本文件对应阶段勾选验收 |

## 四、阶段 1 — 公司内部完美使用

目标：**单租户（`tenant_id = internal`）下，运营可通过反馈持续改进系统，且不影响对话性能。**

### Sprint A — 反馈数据与关联

**目标**：每次 👍👎 可追溯到完整对话上下文。

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| A1 | Feedback 存 SQLite（与 `chat_session` 同库或独立 `feedback.db`） | 可按 `trace_id` / `user_id` / 时间查询 | ✅ |
| A2 | 扩展 `FeedbackRequest`：关联 `trace_id`、`session_id`、`question`、`answer_preview`、`answer_mode` | 向后兼容旧客户端 | ✅ |
| A3 | 写入 feedback 时异步补全 trace 快照（`context_count`、`sources`） | `/chat` 响应时间无 measurable 增加 | ✅ |
| A4 | 保留 `feedback.jsonl` 导出（可选，定时或 on-demand） | 便于离线分析 | ✅ |
| A5 | Admin **反馈列表页**：筛选、分页、跳转 trace | 运营可浏览近 7 天反馈 | ✅ |

**模块建议**：`enterprise_rag/src/feedback_loop/store.py`、`api/feedback_router.py`

**风险缓解**：写入走 `BackgroundTasks` 或线程池；查列表仅 Admin、分页 limit≤50。

---

### Sprint B — Triage 框架（LLM 研判）

**目标**：负反馈自动分类，进入待处理队列。

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| B1 | `feedback_loop/triage.py`：LLM 输出结构化 JSON（`issue_type`、`severity`、`suggested_actions`、`human_review_required`） | 单元测试 mock LLM | ✅ |
| B2 | `issue_type` 枚举：`retrieval_miss`、`hallucination`、`stale_doc`、`prompt`、`tone`、`ok` | 文档化各类型处理策略 | ✅ `issue_types.py` |
| B3 | `feedback_loop/queue.py` 状态机：`pending → triaged → approved → applied → rejected → evaluated` | 非法状态转换拒绝 | ✅ |
| B4 | 后台任务：定时或手动 `POST /admin/feedback/triage` 批量 Triage | **不在** `/chat` 路径执行 | ✅ |
| B5 | Admin **反馈 Inbox**：展示 Triage 结果、severity 排序 | 运营可一键「采纳建议」 | ✅ |

**LLM 提示词**：独立 slot（`prompt_config` 或 `feedback_triage_prompt`），可 Admin 编辑。

**风险缓解**：Triage 失败 → 保持 `pending`；LLM 超时 30s；批大小可配置。

---

### Sprint C — Actuator 与配置版本

**目标**：经审批的动作真正改变系统行为，且可回滚。

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| C1 | Actuator 白名单注册表 | 见下表 |
| C2 | `config_revisions`：提示词 / `kb_min_score` / hybrid 权重等变更留版本号 | 可查看 diff、回滚 |
| C3 | 动作：`add_to_golden` | 高置信 + 审批后写入 `golden.jsonl` |
| C4 | 动作：`propose_reingest` | 生成工单，人工确认后调现有 ingest |
| C5 | 动作：`apply_config_patch` | 仅允许白名单字段；禁止自动删索引 |
| C6 | 配置变更后自动 `invalidate_search_cache` | 检索行为立即一致 |

**Actuator 白名单（阶段 1）**

| 动作 | 自动执行条件 | 默认 |
|------|----------------|------|
| `add_to_golden` | confidence ≥ 0.85 且 rating ≤ 0 | 可自动，可关 |
| `propose_reingest` | 任意 | 仅建议 |
| `apply_config_patch` | 任意 | 必须人工批准 |
| `delete_index` | — | **禁止** |

---

### Sprint D — Evaluate 闭环

**目标**：证明改动有效，避免越改越差。

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| D1 | 变更后触发评测任务（异步）：对比 revision N vs N-1 | Admin 展示指标 delta |
| D2 | 集成 `ragas_scorer` + naive fallback | 无 API Key 时仍可跑 naive |
| D3 | Admin **评测报告页**：faithfulness / overlap / 自定义业务指标 | 可导出 JSON |
| D4 | 每周定时评测（可选 cron / 管理端按钮） | 不阻塞 API |

---

### Sprint E — 租户预留与 API 规范（轻量）

**目标**：为阶段 2 埋线，单租户零行为变化。

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| E1 | `TenantContext` 中间件，默认 `tenant_id=internal` | 全链路可访问、可测 |
| E2 | feedback / session / trace schema 增加 `tenant_id` | 迁移脚本或默认值 |
| E3 | 新路由统一 `/api/v1/...`（旧路由保留别名 deprecate） | OpenAPI 分组清晰 |
| E4 | `FeedbackStore` / `TraceStore` 抽象接口 | 实现 SQLite，文档注明 PG 迁移点 |

---

### Sprint F — Admin 鉴权（内部）

**目标**：管理端不裸奔。

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| F1 | `AdminLayout` auth guard：JWT 或现有 SSO 占位 + 开发态 bypass | 未登录跳转登录 |
| F2 | 角色：`viewer` / `operator` / `admin` | 反馈 inbox、配置、Actuator 分权 |
| F3 | 管理 API 与 Chat API 密钥分离（可选） | `deploy_security.md` 更新 |

## 五、阶段 2 — 外网 SaaS（多租户）

**前置**：阶段 1 在内部稳定运行 ≥ 4 周，且 Actuator + Evaluate 闭环已验证。

### 2.1 三端分离

| 端 | 用户 | 域名示例 | 能力 |
|----|------|----------|------|
| **平台运营端** | 我方 | `ops.example.com` | 租户开通、计费、全局监控、封禁、合规审计 |
| **客户管理端** | 客户管理员 | `{tenant}.admin.example.com` | 入库、用户、本租户配置、反馈 inbox |
| **客户使用端** | 客户终端用户 | `{tenant}.chat.example.com` | 仅 Chat + 反馈 |

### 2.2 租户与安全

| 任务 | 说明 | 验收标准 |
|------|------|----------|
| G1 | 租户级数据隔离：索引、会话、反馈、配置、trace | 跨租户查询返回空或 403 |
| G2 | 鉴权：OAuth2 / API Key per tenant；RBAC | 替换单一 `RAG_API_SECRET` |
| G3 | 配额：QPS、存储、月度 token 上限 | 超限 429 + 指标 |
| G4 | 审计日志：谁改了什么配置 | 保留 ≥ 90 天 |
| G5 | 平台运营 **不可** 默认读客户原文 | 仅聚合指标；合规开关 |

### 2.3 每租户 Feedback Loop

| 任务 | 说明 |
|------|------|
| H1 | golden、config_revisions、inbox **按 tenant 隔离** |
| H2 | Triage 可用平台统一模型，Actuator 作用域限定 tenant |
| H3 | 高风险动作默认 **客户 admin 审批** |

### 2.4 基础设施

| 任务 | 说明 |
|------|------|
| I1 | PostgreSQL（租户、用户、反馈、配置版本） |
| I2 | 向量库 per-tenant 或 collection 前缀 + ACL |
| I3 | Redis 缓存多租户 key 前缀 |
| I4 | `deploy_security.md` + `production_deploy.md` 全量过一遍 |

### 阶段 2 总验收

- [ ] 两租户数据互不可见（渗透测试用例）
- [ ] 客户管理员可独立完成入库 → 对话 → 反馈 → 改进
- [ ] 平台端可关停单租户而不影响其他租户
- [ ] 通过 `CORS`、OpenAPI 关闭、HTTPS、HSTS 生产清单

## 六、执行协议（每步怎么做）

### 6.1 开工前

1. 确认当前分支：`feature/<阶段>-<简述>` 或 `fix/<简述>`
2. 阅读本文档对应 Sprint + **第二节检查清单**
3. 写 1～3 条失败测试（TDD 红）

### 6.2 实现中

1. 热路径与冷路径分离（反馈、Triage、评测 = 冷路径）
2. 新配置项进 `.env.example` + `config.py`，带默认值
3. 日志与 trace 带 `trace_id`、`tenant_id`

### 6.3 合并前

1. `pytest` 相关用例通过
2. 前端改动：`npm test` / 手测 Admin + Chat
3. 若动检索/索引：跑性能基准并记录数字
4. 更新 `README.md` 已实现功能表（若用户可见）
5. 在本文档对应任务打 `[x]`

### 6.4 发布节奏建议

| 节奏 | 内容 |
|------|------|
| 每 Sprint | 可独立上线，灰度给运营试用 |
| 每 2 周 | 跑一轮 Evaluate + 回顾 inbox 积压 |
| 阶段 1 完成 | 内部全员推广 + 收集 30 条以上有效反馈 |
